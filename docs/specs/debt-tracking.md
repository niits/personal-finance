## 1. Introduction

### 1.1 Purpose

This document specifies the requirements for the Debt Tracking feature (Epic 4). Debt tracking allows users to record lending and borrowing relationships, track repayments over time, and see the net impact of debt movements on their overall cash flow.

### 1.2 Scope

The feature covers:

- Recording a new debt (lending or borrowing) by marking a transaction as a debt opening
- Linking and unlinking existing transactions to a debt record
- Recording repayments by creating transactions that reference a debt
- Viewing per-debt repayment history and remaining balance
- Integrating debt cash flows into the home screen income/expense totals
- Excluding debt transactions from the monthly budget pace calculation

### 1.3 Relationship to existing features

Debt tracking is layered on top of the existing transaction system. A debt record is a metadata container; the actual cash movements are stored as regular `transaction` rows with a `debt_id` foreign key. No separate "repayment" entity exists — repayments are transactions.

## 2. Definitions

| Term | Definition |
|------|------------|
| **Debt** | A named financial relationship between the user and another party, representing money owed in either direction. |
| **Lend** | A debt where the user gave money to another party (`type = 'lend'`). The user expects to be paid back. |
| **Borrow** | A debt where the user received money from another party (`type = 'borrow'`). The user owes repayment. |
| **Opening transaction** | The single transaction that initiated the debt — the original cash movement when the loan was made or received. Identified by `debt.opening_transaction_id`. |
| **Repayment transaction** | Any transaction linked to a debt that is not the opening transaction. Represents a partial or full repayment in either direction. |
| **Remaining balance** | `opening_amount − total_repaid`. The amount still outstanding on a debt. |
| **Opening amount** | The `amount` field of the opening transaction. This is the canonical loan principal. |
| **Total repaid** | Sum of `amount` for all repayment transactions linked to a debt. |
| **Settled** | A debt with `status = 'settled'`. Manually set by the user; the system suggests it when `remaining ≤ 0`. |
| **Party** | The name of the other person in the debt relationship (free text, e.g. "Minh", "Ba"). |
| **Link** | The act of setting `transaction.debt_id` to point to a debt record. |
| **Unlink** | The act of setting `transaction.debt_id` to NULL, removing the transaction from a debt. |

## 3. Data Model

### 3.1 Entity Relationship

```
User ──── Debt (1:N)
           │  opening_transaction_id ──┐
           │                           ↓
           └──── Transaction (1:N) ────┘
                  (via transaction.debt_id)
```

Both foreign keys are nullable. The circular reference between `debt.opening_transaction_id` and `transaction.debt_id` is intentional and handled with nullable FKs and a specific insert order (see §7).

### 3.2 `debt` table

```sql
CREATE TABLE debt (
  id                     TEXT    PRIMARY KEY,
  user_id                TEXT    NOT NULL REFERENCES user(id) ON DELETE CASCADE,
  type                   TEXT    NOT NULL CHECK (type IN ('lend', 'borrow')),
  party                  TEXT    NOT NULL,
  note                   TEXT,
  due_date               TEXT,             -- YYYY-MM-DD, nullable
  status                 TEXT    NOT NULL  DEFAULT 'open'
                                   CHECK (status IN ('open', 'settled')),
  opening_transaction_id INTEGER           -- nullable FK
                                   REFERENCES "transaction"(id) ON DELETE SET NULL,
  created_at             TEXT    NOT NULL  DEFAULT (datetime('now'))
);
```

**Field notes:**

| Field | Notes |
|-------|-------|
| `id` | UUID (TEXT), generated by application |
| `type` | `lend` = user gave money; `borrow` = user received money |
| `party` | Free-text name, not a FK — no party entity exists |
| `note` | High-level label for the debt (e.g. "iPhone bị vỡ"). Separate from `transaction.note` |
| `due_date` | Optional. When set, the UI shows an overdue indicator if `due_date < today AND status = 'open'` |
| `status` | Manually set. Not auto-computed. System suggests settling when `remaining ≤ 0` |
| `opening_transaction_id` | Set after the opening transaction is created. NULL during the brief atomic insert window |

**There is no `amount` column.** The principal is derived from `opening_transaction.amount` at read time.

### 3.3 `transaction` table (existing, partial)

Relevant columns:

| Column | Type | Notes |
|--------|------|-------|
| `debt_id` | TEXT | Nullable FK → `debt.id`, `ON DELETE SET NULL` |
| `category_id` | INTEGER | Nullable. Debt transactions have no category |
| `monthly_budget_id` | INTEGER | Nullable. Debt transactions have no budget |
| `type` | TEXT | `expense` or `income` — still required |

The existing CHECK constraint on `transaction`:
```sql
CHECK (
  (type = 'income') OR
  (type = 'expense' AND (monthly_budget_id IS NOT NULL OR debt_id IS NOT NULL))
)
```
This ensures every expense transaction either belongs to a monthly budget or to a debt — not free-floating.

### 3.4 Computed values (not stored)

| Value | Formula |
|-------|---------|
| `opening_amount` | `opening_transaction.amount` |
| `total_repaid` | `SUM(t.amount) WHERE t.debt_id = debt.id AND t.id != debt.opening_transaction_id` |
| `remaining` | `opening_amount − total_repaid` |
| `is_overdue` | `debt.due_date IS NOT NULL AND debt.due_date < current_date AND debt.status = 'open'` |

### 3.5 Cash flow direction by transaction type

| Debt type | Transaction role | Transaction type | Cash meaning |
|-----------|-----------------|-----------------|--------------|
| `lend` | Opening | `expense` | User paid out (gave money) |
| `lend` | Repayment | `income` | User received back |
| `borrow` | Opening | `income` | User received (borrowed) |
| `borrow` | Repayment | `expense` | User paid back |

The transaction `type` is always derived from the debt type and role — the user never chooses it independently for debt transactions.

### 3.6 Migration

Migration `0013_debt_model_revision.sql` drops the `amount` column and adds `opening_transaction_id` and `due_date`:

```sql
-- 0013_debt_model_revision.sql
PRAGMA foreign_keys = OFF;

CREATE TABLE debt_new (
  id                     TEXT    PRIMARY KEY,
  user_id                TEXT    NOT NULL REFERENCES user(id) ON DELETE CASCADE,
  type                   TEXT    NOT NULL CHECK (type IN ('lend', 'borrow')),
  party                  TEXT    NOT NULL,
  note                   TEXT,
  due_date               TEXT,
  status                 TEXT    NOT NULL DEFAULT 'open'
                                   CHECK (status IN ('open', 'settled')),
  opening_transaction_id INTEGER REFERENCES "transaction"(id) ON DELETE SET NULL,
  created_at             TEXT    NOT NULL DEFAULT (datetime('now'))
);

-- Migrate existing rows. opening_transaction_id left NULL — pre-existing
-- debts have no reliable way to identify their original opening transaction.
-- Users may re-link manually if needed.
INSERT INTO debt_new (id, user_id, type, party, note, status, created_at)
  SELECT id, user_id, type, party, note, status, created_at
  FROM debt;

DROP TABLE debt;
ALTER TABLE debt_new RENAME TO debt;

PRAGMA foreign_keys = ON;
```

## 4. Business Rules & Constraints

### 4.1 Debt ownership

| ID | Rule |
|----|------|
| D-01 | A debt belongs to exactly one user (`debt.user_id`). Users can only read and modify their own debts. |
| D-02 | A transaction can belong to at most one debt (`transaction.debt_id` is a single FK, not many-to-many). |
| D-03 | A debt's opening transaction must belong to the same user as the debt. |

### 4.2 Opening transaction

| ID | Rule |
|----|------|
| D-04 | Each debt has at most one opening transaction (`debt.opening_transaction_id`). |
| D-05 | The opening transaction's type must match the debt type: `lend → expense`, `borrow → income`. |
| D-06 | A debt with no opening transaction (`opening_transaction_id IS NULL`) is considered incomplete. It can exist transiently during atomic insert but must not be returned to the UI in that state. |
| D-07 | The opening transaction cannot be unlinked if the debt has any repayment transactions (`total_repaid > 0`). The user must unlink or delete all repayments first. |

### 4.3 Repayment transactions

| ID | Rule |
|----|------|
| D-08 | A repayment transaction's type is the inverse of the opening transaction type: `lend → income repayments`, `borrow → expense repayments`. |
| D-09 | `total_repaid` may exceed `opening_amount` (overpayment is allowed; the UI shows `remaining < 0`). |
| D-10 | Any repayment can be unlinked at any time without restriction. |
| D-11 | Unlinking a repayment does not delete the transaction — it becomes a regular transaction with no debt context. After unlinking, the transaction must satisfy the existing `CHECK` constraint (i.e., it needs a `monthly_budget_id` if it is an `expense`). If this constraint would be violated, the unlink is rejected. |

### 4.4 Status

| ID | Rule |
|----|------|
| D-12 | `status` is set manually by the user. The system never auto-updates it. |
| D-13 | When `remaining ≤ 0`, the UI surfaces a "Tất toán ngay" (settle now) action. Tapping it sets `status = 'settled'`. |
| D-14 | A settled debt is read-only. Adding repayments or linking transactions to a settled debt is not allowed. The user must reopen it first (set `status = 'open'`). |

### 4.5 Linking existing transactions

| ID | Rule |
|----|------|
| D-15 | Only transactions with `debt_id IS NULL` can be linked. |
| D-16 | A transaction being linked as a repayment must have the correct type (see D-08). |
| D-17 | A transaction cannot be linked to a settled debt (see D-14). |

### 4.6 Deletion

| ID | Rule |
|----|------|
| D-18 | Deleting a debt cascades to set `debt_id = NULL` on all linked transactions (via `ON DELETE SET NULL`). The transactions themselves are not deleted. |
| D-19 | Deleting an opening transaction sets `debt.opening_transaction_id = NULL` (via `ON DELETE SET NULL`), leaving the debt incomplete. The UI must surface this state clearly. |

### 4.7 Budget and statistics integration

| ID | Rule |
|----|------|
| D-20 | Debt transactions (`debt_id IS NOT NULL`) are excluded from monthly budget progress calculation. Budget tracks discretionary spending; capital transfers are not spending. |
| D-21 | Debt transactions are included in the home screen monthly income/expense totals (`total_income`, `total_expense`). They represent real cash movements. |
| D-22 | Debt transactions are excluded from category-level breakdowns in the statistics page (they have `category_id IS NULL`). |
| D-23 | Queries that aggregate by category must handle `category_id IS NULL` rows gracefully — skip them, do not crash or show an undefined label. |

## 5. Functional Requirements

### 5.1 Debt management

| ID | Requirement | Priority |
|----|-------------|----------|
| DEBT-01 | Users must be able to create a new debt from the transaction entry form by selecting "Cho vay" or "Đi vay" as transaction context | Must Have |
| DEBT-02 | Creating a new debt requires: party name, and the transaction amount/date/note are inherited from the transaction being entered | Must Have |
| DEBT-03 | Users may optionally set a due date when creating a debt | Should Have |
| DEBT-04 | Users must be able to edit a debt's `party`, `note`, `due_date`, and `status` | Must Have |
| DEBT-05 | Users must not be able to change a debt's `type` (`lend`/`borrow`) after creation | Must Have |
| DEBT-06 | Users must be able to delete a debt. Linked transactions are not deleted; their `debt_id` is cleared | Must Have |
| DEBT-07 | Users must be able to reopen a settled debt (set `status = 'open'`) | Should Have |

### 5.2 Linking & unlinking

| ID | Requirement | Priority |
|----|-------------|----------|
| DEBT-08 | Users must be able to link an existing transaction to a debt as a repayment | Must Have |
| DEBT-09 | Users must be able to unlink a repayment transaction from a debt at any time | Must Have |
| DEBT-10 | Users must not be able to unlink the opening transaction if repayments exist; the system must display a clear error message listing the constraint | Must Have |
| DEBT-11 | Users must be able to link/unlink from both the transaction detail view and the debt detail view | Should Have |

### 5.3 Repayments

| ID | Requirement | Priority |
|----|-------------|----------|
| DEBT-12 | Users must be able to record a repayment by creating a new transaction directly from the debt detail view | Must Have |
| DEBT-13 | The repayment form must show the remaining balance as a hint on the amount field | Should Have |
| DEBT-14 | Users must be able to settle a debt in one action ("Tất toán ngay") when `remaining > 0`, which creates a repayment transaction for the exact remaining amount and sets `status = 'settled'` | Must Have |
| DEBT-15 | When a new repayment amount equals the remaining balance, the system must suggest settling the debt | Should Have |

### 5.4 Debt overview

| ID | Requirement | Priority |
|----|-------------|----------|
| DEBT-16 | The Nợ tab must display two sections: debts the user has lent ("Cho vay") and debts the user has borrowed ("Đi vay") | Must Have |
| DEBT-17 | Each debt card must show: party name, note snippet, progress toward repayment (remaining / opening amount), and creation date | Must Have |
| DEBT-18 | Overdue debts (`is_overdue = true`) must display a visible indicator | Should Have |
| DEBT-19 | Settled debts must be grouped in a collapsed section at the bottom of the list | Must Have |
| DEBT-20 | A summary strip at the top of the Nợ tab must show total outstanding lent and total outstanding borrowed | Must Have |

### 5.5 Debt detail

| ID | Requirement | Priority |
|----|-------------|----------|
| DEBT-21 | The debt detail view must list all linked transactions (opening + repayments) ordered by date | Must Have |
| DEBT-22 | Each linked transaction must show an unlink action | Must Have |
| DEBT-23 | The detail view must display: opening amount, total repaid, remaining balance | Must Have |

### 5.6 Statistics integration

| ID | Requirement | Priority |
|----|-------------|----------|
| DEBT-24 | The home screen header must include debt transactions in monthly income and expense totals | Must Have |
| DEBT-25 | The budget pace bar and remaining budget calculation must exclude debt transactions | Must Have |
| DEBT-26 | The statistics page category breakdown must skip transactions with `category_id IS NULL` without error | Must Have |

## 6. User Flows

### Flow 1 — Record a new loan from the transaction form

```
Home screen → Transaction form
  User enters amount, selects type = "Chi tiêu"
  → Taps "Liên kết nợ" (optional section at bottom of form)
  → Selects "Cho vay mới"
    → "Tên người" field appears (required)
    → "Hạn trả" field appears (optional)
  User fills party name and taps "Lưu"
    → API: POST /api/debts (atomic)
        1. INSERT debt (opening_transaction_id = NULL)
        2. INSERT transaction (type=expense, debt_id=debt.id, category_id=NULL, monthly_budget_id=NULL)
        3. UPDATE debt SET opening_transaction_id = transaction.id
    → Form resets
    → Transaction appears in home screen list with debt badge
    → Debt appears in Nợ tab under "Cho vay"
```

### Flow 2 — Record a repayment from debt detail

```
Nợ tab → Tap debt card → Debt detail
  User taps "Ghi nhận thanh toán"
    → Transaction form opens (pre-configured):
        type = income (auto, derived from lend debt)
        debt_id = this debt (locked)
        amount hint = "còn lại X ₫"
    User fills amount and taps "Lưu"
      → API: POST /api/transactions (with debt_id set)
      → Debt detail updates: total_repaid increases, remaining decreases
      → If remaining ≤ 0: "Tất toán ngay" banner appears
```

### Flow 3 — One-tap settle

```
Debt detail (remaining > 0 or remaining = 0 with status=open)
  User taps "Tất toán ngay"
    → Confirm dialog: "Tất toán [remaining] ₫ từ [party]?"
    → User confirms
      → If remaining > 0:
          API: POST /api/transactions (repayment for exact remaining amount)
      → API: PATCH /api/debts/[id] { status: 'settled' }
      → Debt moves to "Đã tất toán" section in Nợ tab
```

### Flow 4 — Link an existing transaction to a debt

```
Nợ tab → Debt detail
  User taps "Liên kết giao dịch có sẵn"
    → Transaction picker: lists transactions with:
        debt_id IS NULL
        correct type for repayment (income for lend, expense for borrow)
    User selects a transaction
      → API: PATCH /api/transactions/[id]/link { debt_id: "..." }
      → Transaction appears in debt's repayment history
      → Remaining balance updates
```

### Flow 5 — Unlink a repayment

```
Debt detail → Tap transaction row → Tap "Hủy liên kết"
  → Validation: is this the opening transaction?
      No → proceed
      Yes → check if total_repaid > 0
          Yes → Error: "Cần xóa tất cả thanh toán trước khi hủy giao dịch gốc"
          No → proceed
  → Confirm: "Giao dịch này sẽ không còn thuộc khoản nợ. Tiếp tục?"
  → API: DELETE /api/transactions/[id]/link
      → Validates post-unlink transaction still satisfies CHECK constraint
      → Sets transaction.debt_id = NULL
  → Debt detail updates
```

### Flow 6 — Borrow and pay back

```
-- Borrowing --
Home screen → Transaction form
  Amount: 1.000.000, type = "Thu nhập"
  → Taps "Liên kết nợ" → "Đi vay mới"
  → Party: "Ba", no due date
  → Lưu
    → Debt created (type=borrow), opening tx type=income
    → Appears in "Tôi đang nợ" section

-- Paying back --
Nợ tab → "Ba" debt → Tap "Ghi nhận thanh toán"
  → Form pre-configured: type=expense, amount hint "còn lại 1.000.000 ₫"
  → Amount: 1.000.000, date: today
  → Lưu
    → Repayment tx created (type=expense), remaining = 0
    → System suggests "Tất toán ngay"
  → User taps "Tất toán ngay" → PATCH debt status=settled
  → Debt moves to settled section
```

## 7. UI Design

### 7.1 Principles

- **One form for all cash movements.** `TransactionForm` handles new transactions, new debts, repayments, and edits. No separate debt creation or repayment sheet.
- **Full-screen modal.** The form slides up to full viewport height. A drag handle is retained for muscle memory, but close is via an explicit ✕ button. The save action ("Lưu") lives in the nav bar top-right.
- **Context-driven rendering.** The form receives a `mode` prop that drives which fields are visible, locked, or pre-filled. No scattered boolean flags in JSX.
- **Repayment = transaction.** Recording a repayment opens `TransactionForm` in `repayment` mode — pre-filled amount, locked type, locked debt chip, no category.

### 7.2 TransactionForm modes

```typescript
type TransactionFormMode =
  | { kind: "create" }
  | { kind: "create-debt-open" }                       // FAB on Nợ tab — debt section pre-expanded
  | { kind: "repayment"; debt: DebtWithRepayments }    // from Debt Detail → Ghi nhận thanh toán
  | { kind: "edit"; transaction: EditTransaction };
```

| Mode | Title | Type segmented | Category | Debt section | Custom budgets |
|------|-------|---------------|----------|-------------|----------------|
| `create` | Giao dịch mới | Editable | Shown | Collapsed, optional | Shown (expense) |
| `create-debt-open` | Giao dịch mới | Editable | Hidden when debt selected | Pre-expanded | Hidden when debt selected |
| `repayment` | Ghi nhận thanh toán | Hidden (locked) | Hidden | Locked chip, no options | Hidden |
| `edit` (normal tx) | Sửa giao dịch | Editable | Shown | Collapsed, optional | Shown (expense) |
| `edit` (debt tx) | Sửa giao dịch | Hidden (locked) | Hidden | Locked chip + unlink action | Hidden |

## 8. API Contract

### 8.1 `POST /api/debts`

Creates a new debt and its opening transaction atomically.

**Request body:**

```typescript
{
  type: "lend" | "borrow";
  party: string;              // non-empty
  note?: string;
  due_date?: string;          // YYYY-MM-DD
  // Opening transaction fields
  amount: number;             // positive integer, VND
  date: string;               // YYYY-MM-DD
  transaction_note?: string;  // separate from debt.note
}
```

**Response `201`:**

```typescript
{
  debt: DebtWithRepayments;
}
```

**Atomic insert sequence:**

```
1. INSERT debt (opening_transaction_id = NULL)
2. INSERT transaction (
     type = lend→expense | borrow→income,
     amount, date, note = transaction_note,
     debt_id = debt.id,
     category_id = NULL,
     monthly_budget_id = NULL
   )
3. UPDATE debt SET opening_transaction_id = transaction.id
```

All three statements run in a single D1 batch.

---

### 8.2 `GET /api/debts`

Returns all debts for the authenticated user, grouped.

**Response `200`:**

```typescript
{
  lending:   DebtWithRepayments[];  // type=lend, status=open
  borrowing: DebtWithRepayments[];  // type=borrow, status=open
  settled:   DebtWithRepayments[];  // status=settled, both types
}
```

**`DebtWithRepayments` type:**

```typescript
{
  id: string;
  type: "lend" | "borrow";
  party: string;
  note: string | null;
  due_date: string | null;
  status: "open" | "settled";
  opening_transaction_id: number | null;
  created_at: string;
  // Computed
  opening_amount: number;         // opening_transaction.amount, 0 if no opening tx
  total_repaid: number;
  remaining: number;              // opening_amount - total_repaid
  is_overdue: boolean;
  transactions: LinkedTransaction[];
}

type LinkedTransaction = {
  id: number;
  type: "expense" | "income";
  amount: number;
  date: string;
  note: string | null;
  is_opening: boolean;           // id === debt.opening_transaction_id
}
```

---

### 8.3 `GET /api/debts/[id]`

Returns a single debt with full transaction list. Same `DebtWithRepayments` shape.

**Error `404`** if debt does not belong to authenticated user.

---

### 8.4 `PATCH /api/debts/[id]`

Updates editable debt fields.

**Request body (all optional):**

```typescript
{
  party?: string;
  note?: string | null;
  due_date?: string | null;
  status?: "open" | "settled";
}
```

**Not allowed via this endpoint:** `type`, `opening_transaction_id`.

**Response `200`:** Updated `DebtWithRepayments`.

---

### 8.5 `DELETE /api/debts/[id]`

Deletes a debt. All linked transactions have `debt_id` set to NULL (via FK cascade). Transactions are not deleted.

**Response `204`**.

---

### 8.6 `PATCH /api/transactions/[id]/link`

Links an existing transaction to a debt as a repayment.

**Request body:**

```typescript
{
  debt_id: string;
}
```

**Validation (in order):**

1. Transaction belongs to authenticated user
2. Transaction currently has `debt_id IS NULL`
3. Target debt belongs to authenticated user
4. Target debt `status = 'open'`
5. Transaction type matches expected repayment direction (lend → income, borrow → expense)

**Response `200`:** Updated transaction row.

**Error `409`** with `reason` field for each validation failure.

---

### 8.7 `DELETE /api/transactions/[id]/link`

Unlinks a transaction from its debt.

**Validation (in order):**

1. Transaction belongs to authenticated user
2. Transaction has `debt_id IS NOT NULL`
3. If this transaction is the opening transaction (`id = debt.opening_transaction_id`):
   - Check `total_repaid = 0`. If not → `409` with `reason: "opening_has_repayments"`
4. Post-unlink, validate the transaction would still satisfy the CHECK constraint:
   - If `type = 'expense'` and `monthly_budget_id IS NULL` → `409` with `reason: "expense_requires_budget"`

**Response `200`:** Updated transaction row (with `debt_id: null`).

---

### 8.8 Removed endpoint

| Endpoint | Reason |
|----------|--------|
| `POST /api/debts/[id]/repayments` | Replaced by `POST /api/transactions` (with `debt_id` in body) + `PATCH .../link` pattern |

New repayments are created via the standard `POST /api/transactions` endpoint with `debt_id` in the request body. No separate repayment endpoint is needed.

## 9. Non-Functional Requirements

| ID | Requirement |
|----|-------------|
| NFR-D01 | All debt API endpoints must authenticate the session and scope all queries to `user_id` |
| NFR-D02 | The atomic debt creation (§7.1 insert sequence) must use D1 batch API — no partial writes |
| NFR-D03 | `GET /api/debts` must return within 500ms for up to 100 debts with full transaction enrichment |
| NFR-D04 | The Nợ tab, debt cards, and debt detail view must be responsive on 375px–430px (iPhone) viewports |
| NFR-D05 | All amounts displayed in the debt UI must use vi-VN locale formatting (`.` thousands separator, `₫` suffix) |

---

## 10. Out of Scope

| Item | Notes |
|------|-------|
| Multi-party debts | Each debt has exactly one party |
| Debt categories | Debt transactions have no category |
| Interest calculation | No interest tracking |
| Reminders / push notifications for overdue debts | Out of scope for this epic |
| Splitting a debt across multiple people | Not supported |
| Importing debts from external sources | Not supported |
| Currency other than VND | Not supported |
